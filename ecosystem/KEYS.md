# Sandbox keys — one map

Every private key in the sandbox lives in **one place**: `trusted-list/secrets/` (offline, gitignored). This
file is the committed map of what each key is, what it signs, and where it's deployed. For live fingerprints
and cross-checks, run:

```bash
cd ecosystem/trusted-list && node tools/keys-status.mjs
```

## The hierarchy

```
Scheme Operator (SO)  ──JAdES-signs──▶  the Trusted Lists themselves
  (root of the scheme)                   published at trusted-list.vercel.app/tl/
                                         SO cert downloadable at /tl/scheme-operator.pem
        │  each list publishes a CA as a trust anchor:
        ├─ Wallet Providers list   ▶ WP CA          ◀── WUAs (wallet attestations) chain here
        ├─ PID Issuers list        ▶ PID Issuer CA  ◀── PID credentials chain here
        ├─ Attestation list        ▶ Attestation CA ◀── mDL chains here
        └─ Registrar list          ▶ Registrar CA   (public cert only, from registrar-be)

  Each issuing CA signs a leaf "Document Signer" that the service actually signs with:
        WP CA          ▶ WUA Signer          (wallet-provider signs wallet attestations)
        PID Issuer CA  ▶ PID Document Signer  (issuer signs PID: SD-JWT x5c / mdoc x5chain)
        Attestation CA ▶ mDL Document Signer  (issuer signs mDL mdoc)
```

**SO signs the *lists*. CAs anchor the *credentials*.** A verifier: download + pin the SO cert → verify a
list's signature → trust the CAs on it → verify a credential chains to one of those CAs.

## The keys (in `trusted-list/secrets/`)

| Key file | Role | Chains to | Published | Deployed to |
| --- | --- | --- | --- | --- |
| `so-keystore.json` | **Scheme Operator** (signs every list) | self-signed root | `/tl/scheme-operator.pem` | offline (signing is local) |
| `wp-keystore.json` → `caCertPem` | **Wallet Provider CA** | self-signed root | `config/certs/wp-ca.pem` → wallet-providers list | — |
| `wp-keystore.json` → `certPem` | WUA Signer (ETSI 119 412-6 `id-etsi-qct-wal`) | WP CA | — | AWS `hopae/dev/wp/app-secrets` → `wp-api` |
| `pid-issuer-ca.json` | **PID Issuer CA** | self-signed root | `config/certs/pid-issuer-ca.pem` → pid-issuers list | — |
| `pid-signer.json` | PID Document Signer | PID Issuer CA | — | issuer env `ISSUER_PID_SIGNER` |
| `attestation-ca.json` | **Attestation/mDL CA** | self-signed root | `config/certs/attestation-ca.pem` → attestation list | — |
| `mdl-signer.json` | mDL Document Signer | Attestation CA | — | issuer env `ISSUER_MDL_SIGNER` |

Registrar CA has no private key here — it's the public cert reused from `registrar-be` (`config/certs/registrar-ca.pem`).

## Generating / rotating

All generators write into `trusted-list/secrets/` and print/output what to deploy:

```bash
cd ecosystem/trusted-list
node tools/gen-so-keystore.mjs                                  # SO root
node tools/gen-issuer-ca.mjs pid-issuer-ca  "PID Issuer CA"     # an issuing CA
node tools/gen-signer.mjs pid-issuer-ca pid-signer "PID Document Signer"  # a Document Signer from a CA
# the Wallet Provider keystore is generated by wallet-provider/tools/gen-keystore.mjs
```

After changing a **CA**: update its `config/certs/*.pem`, `npm run gen:tl`, commit + push (Vercel republishes
the list — **the anchor must be live before the consumer redeploys**), then redeploy the consumer (WP secret
/ issuer env) to the new keys.

## Notes

- `~/wp-keystore-dev.txt` is a **stale** earlier WP keystore (superseded) — ignore/delete it; the live WP CA
  is `wp-keystore.json` here (also in AWS + the published Wallet Providers list).
- These are **sandbox dev keys**. For anything real, keep roots offline / in an HSM/KMS and never on disk.
